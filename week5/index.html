Errors, Exceptions, and Warnings

Errors

  • System error ― Program experiences problems with the system or external device it is working with.
  • Programmer error ― Typos, incorrect syntax, bad logic, etc.
  • User error ― The program can't handle a user request or input.

Exceptions

  • An exception is an error which returns a value. The application can be programmed to handle the error based on its identifier.
  • A Stack Trace is produced by an exception. It is a way to track the error based on functions or method calls

Warnings

  • Warnings happen when there is a minor error in the code (minor meaning: not serious enough to cause program failure).
  • Sometimes errors will be administered via command line, console, GUI, or another method. This all depends on the environment where the program runs.

Testing and Debugging: Results

  • Obviously, the goal is to have perfectly executing code, but there should always be appropriate error handling to alert the user/programmer in case of error.
  • The textbook suggests making the code fail "loudly".

Strict Mode

  • Strict mode creates more frequent and srict warnings and exceptions.
  • This is especially good for development, as it throws exceptions for "sloppy" code that could cause errors later.
  • In order to start strict mode, at the beginning of the file, type: 'use strict'
  • Linting Tools are extra tools which highlight and help get rid of/edit sloppy code.

Feature Detection

  • Features are obviously constantly changing... this includes where features are/are not available, when they can be incorporated, etc.
  • In order to incorporate feature detection in your code, use 'if' statements
  • The example from the book uses the 'window.' method to call forth the targeted api before activating a feature. There are other ways to reference the API, but what really matters is that the feature is contained in the 'if' statement.

In-Browser Debugging

  • ALERT
    • An Alert 'alert()' is used to stop the code from running util the 'okay' signal is given by the programmer.
    • It creates breakpoints and can display explanatory messages.
  • THE CONSOLE
    • The console object allows for the 'bookmarking' of code.
    • In the console object, you can manipulate parts of the code without stopping the program (variables, etc).
  • PRE-INSTALLED BROWSER TOOLS
    • Most modern browsers have debuggers built in. Some of these include:
      • Firefox
      • Edge
      • Chrome
      • Safari
    • In these debuggers, you can set breakpoints, see error messages, hover over lines to see more detailed descriptions, etc.

Error Objects

  • Error objects are either automatically created by the system, or manually created for later throws by the programmer.
  • Use the 'new' method to invoke a new Error.
  • SPECIFIC ERROR TYPES
    • Error - Regular, everyday generic error.
    • EvalError - NOT IN CURRENT USE!!! Was used for the 'eval()' function.
    • RangeError - Number is outside of preset range.
    • ReferenceError - Reference was made to an object, function, method, variable, etc. that does not exist.
    • SyntaxError - Error in code's syntax.
    • TypeError - Mixmatch in value type (ex. Number type expected, string provided).
    • URIError - URI problems (encoding, decoding).
    • InternalError - Problems with JavaScript engine. This is a non-standard error.
  • These can all have custom messages.
  • THROWING EXCEPTIONS - Use 'throw' statement. Write a catch block to activate errors when faced.

Exception Handling

  • When code is suspect to an exception (custom or automatic), you include a try block.
  • The try block should include the code in question.
  • Directly after the try block, the catch block is placed
  • The catch block contains the 'backup plan' (code that will be excecuted instead of place of the failed try block.

Tests

  • Tests are written in order to catch errors before they have the chance to become them.
  • I would consider data validation to fall under the category, as well as regular functions that can check calculations and/or return booleans.
  • TEST DRIVEN DEVELOPMENT - Writing tests before writing implementation code. The strategy is best handled in chunks.
    • Workflow:
      • Write tests
      • Write code to compare to tests
      • Refactor code
      • Test refactored code
      • Continue/repeat
  • Testing Frameworks - Some testing frameworks (like Jest) exist for the purpose of having pre-built tests. This is nice for times when writing your own custom tests is too time consuming or complicated.
